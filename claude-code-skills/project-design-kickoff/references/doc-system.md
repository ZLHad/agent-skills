# Doc 体系（Level 2/3）

复杂项目的开发规范应该**切分成若干份解耦的 Doc**，按清晰的依赖顺序产出——避免"一份巨型 README 被所有人改"的混乱。

**但"切几份、叫什么"完全取决于项目类型**。本文档传递**切分思维 + 几种常见项目类型的参考**，不是一份固定清单。

---

## 核心思维：怎么切分 Doc

### 一份合格的 Doc 具备 3 个特征

1. **回答一个单一问题**（"系统有哪些功能" / "数据库长什么样" / "API 怎么调"），不混杂
2. **有明确的上下游**（依赖谁、被谁依赖），改动影响范围可预测
3. **有独立的状态**（🔴 待撰写 / 🟡 骨架 / 🟢 完成），不和其他 Doc 绑死

### 切分的起点：项目的"契约边界"在哪

一个项目通常有几个关键的**契约边界**——不同模块/人/阶段之间需要达成一致的"接口":

- **业务 ↔ 技术**：系统要干什么 → Feature / Goals / Scope Doc
- **模块 ↔ 模块**：模块间传递的数据/调用 → Data / API / Interface Doc
- **系统 ↔ 外部**：数据源、第三方 API、硬件接口 → Integration / Source Doc
- **开发 ↔ 部署**：依赖、环境、流程 → Engineering / Ops Doc
- **用户 ↔ 系统**：交互界面、工作流 → UI / UX / Workflow Doc

**识别你的项目有哪些契约边界**，每个边界一份 Doc，就是合理的切分。

## 常见项目类型的 Doc 切分示例

以下是**参考示例**，不是必须照搬——按项目实际契约边界调整。

### A. Web 应用 / 前后端分离系统

契约边界多，Doc 切得细：

```
01 Feature Spec         功能清单（系统做什么）
02 Data Schemas         跨模块共享的数据结构
03 Database Schema      持久化表结构
04 API Spec             HTTP 端点
05 UI Wireframes        页面布局
06 Engineering Spec     依赖、Git、部署
```

### B. Python 库 / SDK

契约边界主要是对外 API，Doc 很少：

```
01 API Reference        公开接口和使用示例
02 Design Notes         核心设计决策
03 Contributing         贡献和开发规范
```

### C. ML / 训练系统

契约边界是数据、模型、实验：

```
01 Research Goals       解决什么问题、评估标准
02 Dataset Spec         数据来源、清洗、划分
03 Model Spec           模型结构、损失、超参
04 Training Spec        训练流程、checkpoint、日志
05 Experiment Log       每次实验的结果对照
```

### D. Agent / LLM 应用

契约边界综合（数据 + LLM + 业务逻辑 + 可能有前端）：

```
01 Feature Spec         功能清单
02 Data Schemas         业务对象
03 Database Schema      持久化（如有）
04 API Spec             对外接口（如有）
05 Data Sources         外部数据 / 知识来源
06 Agent Specs          Agent 角色、Prompt、Tool
07 UI                   前端（如有）
08 Engineering Spec     依赖和部署
```

这是**最复杂的情况**——融合了多种契约边界。

### E. CLI 工具

```
01 Feature Spec         命令列表、参数、输出
02 Engineering Spec     依赖、打包、分发
```

### F. 数据科学探索项目

契约边界不是接口而是**研究过程**：

```
01 Research Question    问题定义、背景
02 EDA Notes            探索性数据分析
03 Analysis Report      最终结论和可复现步骤
```

### G. 嵌入式 / 硬件相关

```
01 Hardware Interface   硬件接口定义
02 Driver Spec          驱动接口和时序
03 Firmware Design      固件架构
04 Test Plan            测试方案
```

## 依赖关系的通用原则

不管切成几份，以下原则通用：

1. **功能/研究目标在最上游**——先回答"做什么"再回答"怎么做"
2. **数据契约在中间底层**——定义好数据形态，其他 Doc 才能引用
3. **外部接口（API / 用户界面 / 数据源）可以并行**——因为它们分别面向不同角色
4. **工程/部署放最后**——等业务逻辑稳定再做
5. **Doc 是活的，不是一次写完**——跟着开发迭代，状态从 🔴 → 🟡 → 🟢

## Doc 状态标记（通用）

每份 Doc 首部用 admonition 标注：

```markdown
!!! warning "文档状态：骨架版 🟡"
    本文档当前阶段——{范围内} / {范围外} / {演进方式}
```

三档：

- 🔴 **待撰写**：只有 placeholder
- 🟡 **骨架版 / 框架版**：结构和设计原则确定，字段细节待补
- 🟢 **完成**：可作为契约被其他模块 depend on

---

## Placeholder 模板（通用）

新建 Doc 时最小骨架：

```markdown
# {NN} · {文档名}

!!! warning "状态：待撰写"
    依赖：{上游 Doc}

## 文档目的

{一句话说明这份 Doc 回答什么问题}

## 待填充内容

- [ ] ...
- [ ] ...
```

## 两条关键原则

**1. Doc 数量由契约边界决定，不要凑数**

"听说成熟项目要 8 份 Doc 所以我也写 8 份"是反模式。一个纯 Python 库写 8 份 Doc 里有 5 份空文件，比只写 2-3 份实打实的差多了。

**2. Doc 是对话的产物，不追求一次写完**

每次设计讨论的结论落到 Doc 里，状态从 🔴 升到 🟡 再升到 🟢 是正常路径。"Doc 一直是 🔴"或者"一次把 8 份都写到 🟢 才开始开工"都是问题信号。

---

## 怎么决定你的项目切几份

**Level 2/3 初始化时的决策路径**：

1. 列出项目的**契约边界**（参考上方"核心思维"）
2. 每个边界一份 Doc，命名根据项目类型调整
3. 初始化阶段只给每份 Doc **一个 placeholder**（`!!! warning 待撰写`）
4. 随讨论迭代升级到 🟡 / 🟢
5. 如果某份 Doc 写到一半发现"其实不需要这份"，合并或删除它也完全 OK

**不要在初始化阶段强行套 8 份**。小到 2 份，大到 10 份，都是合理的。
