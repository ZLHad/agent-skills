# ReferenceItems 管理规范（Level 3）

复杂项目需要系统化管理**外部参考资料**——开源项目源码、论文、竞品分析、前期讨论等。目标是让这些"不是本项目代码但启发本项目"的资料**可追溯、可对比、可借鉴**。

## 目录结构

```
项目根/
├── ReferenceItems/              ← 原始素材（浅克隆 / PDF / MD 等）
│   ├── <项目A>/                  ← 外部开源项目源码（git clone --depth 1）
│   ├── <项目B>/
│   ├── Papers/                  ← 论文 PDF
│   └── ReferenceDiscussion/     ← 前期讨论记录
│
└── doc/docs/references/         ← 项目对参考资料的分析文档
    ├── index.md                 ← 总索引 + 分层借鉴策略表
    ├── <项目A>.md               ← 每个重要参考的独立分析文档
    ├── <项目B>.md
    └── <主题>.md                 ← 跨项目的专题横向对比
```

## 原始素材下载约定

- **浅克隆**：`git clone --depth 1 <url>` 节省磁盘和网络
- **子模块不自动初始化**：节省时间，需要时再 `git submodule update --init <path>`
- **LICENSE 检查**：下载前确认许可证兼容
- **大小控制**：单个参考项目 > 200 MB 时考虑只拉关键目录（`git sparse-checkout`）

## 每份参考文档的结构（doc/docs/references/<项目>.md）

推荐统一结构，便于横向对比：

```markdown
# <项目名>

!!! abstract "一句话定位"
    {是什么 / 不是什么 / 对我们的价值}

| 项 | 内容 |
|---|---|
| GitHub / 官网 | ... |
| Stars / 用户量 | ... |
| License | ... |
| 语言/技术栈 | ... |
| 本地位置 | `ReferenceItems/<xxx>/` |
| 关键模块 | ... |

## 一、是什么

{核心定位，和本项目的关系}

## 二、值得深读的子系统

### 2.1 ...
{每个值得学的模块，重点不是"它有什么"，而是"对我们有什么借鉴价值"}

## 三、借鉴策略

| 层 | 借鉴方式 | 优先级 |
|---|---------|-------|
| ... | ... | ... |

## 四、不借鉴的部分

{明确列出不跟的，避免被大项目带偏}
```

## 分层借鉴策略表（references/index.md 的核心）

这是**Level 3 最重要的表**——告诉自己"每个层我借鉴哪个参考、借鉴方式是什么"：

```markdown
## 分层借鉴策略

| 层 | 借鉴方式 | 来源 |
|----|---------|------|
| 数据层脚手架 | 代码基本复刻 | <项目A> |
| 核心算法 | 借鉴思路自研 | <项目B> |
| 前端组件 | npm 依赖直接用 | <项目C> |
| Agent 编排 | 架构模仿 | <项目D> |
| {完全自主} | — | — |
```

**三种借鉴方式的区别**：

1. **代码基本复刻**：抄过来改一改就用（UI 脚手架、工具函数）
2. **借鉴思路自研**：学架构和设计模式，自己重写（核心业务逻辑、算法）
3. **直接作为依赖**：npm install / pip install 直接引用（UI 组件库、工具库）

## 横向对比专题（可选）

当多个参考项目在同一维度有不同方案时，建一份"XX 实现模式" 专题文档——横向对比所有方案，提炼本项目的选择。

例子：4 个项目的"LLM 调用模式"对比 → 提炼本项目的混合策略。

## 两条关键原则

**1. 下载原始素材不等于要学它全部**——每个参考项目只看"对我们有价值的 3-5 个点"，明确"不学什么"同样重要

**2. 借鉴策略是"活的"**——随着项目深入，某个参考可能从"架构参考"变成"局部 API 参考"。策略表要随时更新，不要一次定死

## 一个警示

**不要堆一堆参考项目就心安了**——我见过项目 `ReferenceItems/` 下 20 个 repo，但 `references/` 下只有 3 份分析文档，借鉴策略表从来没填过。这样的参考资料等同于噪声。

下载一个 → 写一份分析 → 更新借鉴策略表 → 决定要不要留着 → 才算完成一次参考项目的纳入。
