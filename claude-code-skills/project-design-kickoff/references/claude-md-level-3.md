## 协作模式：文档双轨制

本项目维护两套互补的文档：

- **DesignNotes/** — 会话讨论记录，按 session 编号追加。记录"为什么"和"怎么得出结论的"，**不修改历史**
- **doc/** — MkDocs 正式文档站点。按主题结构化，**反映最新状态**，记录"是什么"和"怎么做"

重大决策同时在两处体现：DesignNotes 记过程，doc 记结论。

### DesignNotes/ 的维护

1. 每次会话产出一份文档：`日期_session-NN_主题.md`
2. 讨论过程中有价值的想法和决策即时写入
3. 新 session 可以"修正"旧 session 的结论，但**不改旧文**
4. `DesignNotes/README.md` 是索引，每次新增 session 要更新
5. **会话末尾主动写 session doc 的触发条件**（满足任一即写）：
   - 本次对话产生了重大设计决策或认知修正
   - 完成了重要功能、模块的修改或讨论
   - 用户主动说"记录一下这次会话"

   满足条件时主动创建 session 文档、更新 README 索引、创建 `discussion/sessions/` 符号链接、更新 mkdocs.yml nav。

### doc/ MkDocs 站点的维护

站点结构：`overview/` + `development/` + `architecture/` + `references/` + `discussion/`

本地运行：`cd doc && mkdocs serve` → http://127.0.0.1:8000

**核心原则**：
- doc/ 下反映**最新决策状态**，不留"session-N 修正"之类的过程元信息
- 战略级决策变化时，同步更新 `overview/` 和 `architecture/` 下相关文档
- 新增参考项目时，在 `references/` 下新建文档并更新借鉴策略
- `discussion/sessions/` 是指向 `DesignNotes/` 的符号链接
- 每次结构性修改后验证 `mkdocs build --strict` 干净通过

### Doc 状态标记约定

- 🔴 **待撰写**：placeholder
- 🟡 **骨架版 / 框架版**：结构定了，字段待补
- 🟢 **完成**：可作为契约

### 跨 session 的认知升级

重大战略决策同步更新 **project memory**——DesignNotes 记过程，memory 记结论，doc 记最新状态。

### 每次会话开始时

1. memory 自动加载
2. 读最新的 session 文档
3. 按任务需要读 doc/ 下的相关章节

## 学术论文与参考资料

- 参考资料和外部项目放在 `ReferenceItems/` 下，**不是本项目代码**，仅供参考和借鉴
- 每个值得深读的参考项目在 `doc/docs/references/` 下建一份分析文档
- 借鉴策略按"哪一层借鉴什么"组织，见 `doc/docs/references/index.md`
- 引用论文时标注年份和会议/期刊

## 语言与风格

- {语言偏好}
- 设计文档要**有观点、有判断**，不要面面俱到地罗列
- 代码和技术方案用**段落式说明**，不堆 bullet point
- 正式文档（doc/ 下）避免 "session-N 修正"之类的元信息噪音——讨论过程留在 DesignNotes/ 里

## 需要特别注意的约束

（项目特定的约束，讨论过程中逐步填充。常见维度：）

1. **核心业务逻辑框架无关**：外部重量级依赖（Web 框架 / 数据库 / ML 库等）通过**薄封装层**隔离，上层业务代码不直接 import。详见 `doc/docs/architecture/layers.md`
2. **垂直切片开发**：每周/迭代都要有端到端可用的成果，不要水平堆叠（先做完所有底层再做上层的模式容易踩雷）
3. **配置驱动，不硬编码**：会变的参数放配置文件，代码读配置
4. **其他领域特定约束**：...

## 隐私与称呼（可选，按项目需要启用）

1. **用户昵称**：所有项目文档、代码注释、DesignNotes、session 一律使用"{昵称}"（不使用真实姓名）
2. 已有文档的真实姓名引用属于隐私消毒的合理例外，可追溯修改
3. 外部路径（如 `/Users/xxx/`）不属于文档内容，保持不变

## 架构扩展性预留（可选，按项目需要启用）

项目采用**轻量自研**方案起步，**不内嵌**重量级外部框架的完整运行时。但接口设计必须为**未来导入外部架构源代码或扩展生态**预留能力。

核心原则：

1. **关键组件配置化**——核心业务单元是配置驱动的"能力单元"，不是硬编码的类
2. **外部调用走统一协议**——不直接 import 外部函数，全部通过抽象接口发起；底层是本地实现 / 外部系统对此透明
3. **adapters/ 目录预留**——即使 Phase 1 不实现任何 adapter，也要预留目录 + README 说明扩展方向
4. **多用户/多环境预留**（如适用）——从 Day 1 就加 user_id / env 字段，单用户/单环境场景默认值填充，未来扩展不动底层

**底线**：实现得轻一点不要紧，**但接口不能把路堵死**。任何"引入外部生态"的动作都应该是"加文件"而不是"改文件"。

## 使用说明

- **关键认知**的 3-5 条在初始化时可能只有 1-2 条，随讨论逐步补齐
- **隐私与称呼**和**架构扩展性预留**两章是**可选**的——根据项目实际需要决定要不要在初始化时就写入
- 如果项目有**阶段划分**（Phase 1/2/3 等），在"项目概述"章节后加一个独立的"阶段划分"说明
- **Doc 体系**具体选哪几份、架构扩展性具体怎么落，分别参考 `doc-system.md` 和 `architecture-patterns.md`
